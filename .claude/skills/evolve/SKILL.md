# /evolve — Harness 自进化

**目标**：把这次会话里踩过的坑、走过的弯路、发现的好模式，变成下次不会再犯的规则，写进 skill 文件，让整个 harness 越来越健壮。

**不是总结报告，是代码级改动。** 每次 `/evolve` 执行完，至少有一个 SKILL.md 文件被实际修改。

---

## 触发时机

- `project-flow` Step 8 结束、用户确认"满意"之后
- `everything` Phase 8 结束、用户确认"OK"之后
- 用户主动输入 `/evolve`

**不要在以下情况触发**：
- 用户还在调整需求中
- 部署失败未解决
- 用户明确说"先不管进化，做下一个需求"

---

## 执行流程

### Step 1 — 收集本次会话的事实

并行执行：

```bash
# 今天的提交记录
git log --oneline --since="12 hours ago"

# 今天新增的 dev-notes
ls -t doc/dev-notes/ | head -5

# 今天 CHANGELOG 里的新条目（取最新的一段）
head -60 doc/CHANGELOG.md
```

同时在会话上下文中回顾（不需要执行命令，直接从对话历史提取）：
- 哪些步骤被重试了（第一次失败、第二次才过）
- 哪些假设被用户纠正了（用户说"不对，我想要的是…"）
- 哪些错误出现后需要额外的修复步骤
- 哪些步骤比预期花了更多时间

### Step 2 — 自我反思：提炼教训

对每一个"弯路"，问以下三个问题：

1. **根本原因是什么**？（是假设错了？是步骤遗漏了？是边界条件没考虑？）
2. **哪条 skill 规则如果存在，就能提前避免这个问题**？
3. **这个规则是新的，还是已经存在但没被遵守**？

同时提炼"正向模式"：
- 哪个做法顺利跑通，值得固化为默认行为？
- 哪个检查步骤发现了问题，应该提前到更早做？

**筛选标准 — 只保留值得写进 skill 的教训**：
- ✅ 会在其他项目再次出现的通用问题
- ✅ 不写进规则就会再犯的陷阱
- ✅ 节省超过 5 分钟的优化
- ❌ 一次性的业务特定问题（如"这个项目的表名叫 X"）
- ❌ 已经写在 skill 里的内容（先检查，再决定是否写）
- ❌ 纯代码层面的修复（修 bug ≠ 改 skill）

### Step 3 — 定位更新目标

把每条教训分配到最相关的文件：

| 教训类型 | 更新目标 |
|---------|---------|
| 部署前应做的检查 | `deploy/SKILL.md` |
| 需求分析遗漏的问题 | `project-flow/SKILL.md` Step 1 或 `everything/SKILL.md` Phase 1 |
| 本地验证遗漏的检查 | `project-flow/SKILL.md` Step 4 |
| 代码规范/架构模式 | `CLAUDE.md` 的「避免的反模式」或「常见场景速查」 |
| 非技术用户引导方式 | `everything/SKILL.md` |
| 多个 skill 都要改 | 分别写，不要合并 |

### Step 4 — 展示更新摘要（轻量告知，自动写入）

用一行摘要告知用户即将发生的改动，**不需要等待确认**：

```
记录本次经验：[deploy] +1条部署前检查，[project-flow] +1条需求澄清 → 10秒后自动写入，如有异议请告知。
```

**等待 10 秒**（给用户反对的机会）：
- 用户说"不对"/"别写"/"等一下" → 停止，展示详细内容让用户逐条确认后再写
- 用户说"好"/"可以" → 立即执行 Step 5，不再等待
- 无响应 → 10 秒后直接执行 Step 5

**何时改为要求明确确认**（例外情况）：
- 本次改动 ≥ 3 个 skill 文件（改动范围较大）
- 用户在本次会话里曾明确说某个做法"我不喜欢"（避免把用户不认同的模式写进去）
- 用户主动输入 `/evolve`（说明用户有意识地想审查，则展示详细清单）

### Step 5 — 写入 skill 更新

**写入原则**：
- 找到最相关的位置插入，不要放在文件末尾堆积
- 每条教训 ≤ 3 行：一行规则 + 一行原因（可选）
- 在对应 skill 文件里找「已知陷阱」或「Known Gotchas」section，没有就新建
- 如果是对已有规则的强化，直接修改原有规则，不要追加

**「已知陷阱」section 格式**（在对应 SKILL.md 末尾，如果不存在就创建）：

```markdown
## 已知陷阱（由 /evolve 持续更新）

<!-- 每条格式：- [YYYY-MM-DD] 规则描述。原因：XXX -->
- [2026-04-17] 部署前先确认 .env.docker.prod 中没有 placeholder 值。原因：默认密码在容器启动时会静默失败，不报明显错误。
```

**同时更新 `doc/harness-evolution.md`**（进化日志，见 Step 6）。

### Step 6 — 写入进化日志

在 `doc/harness-evolution.md` 的**顶部**插入新条目：

```markdown
## [YYYY-MM-DD] <需求一句话摘要>

**需求**: <用户原始描述，一句话>
**结果**: 成功上线 / 部分完成 / 失败

**质量指标**:
- 重试次数: N（指某个步骤第一次失败需要修复后重试）
- 用户纠偏次数: N（指用户说"不对，我想要的是…"的次数）
- 首次部署成功: 是 / 否

**问题 → 教训 → 写入位置**:
| 遇到的问题 | 根本原因 | 新增规则 | 更新了哪个文件 |
|-----------|---------|---------|-------------|
| TypeScript 构建报错 | 缺少 type import | 部署前先跑 tsc | project-flow Step 4 |

**正向模式（值得保留的做法）**:
- <做了什么，为什么好>

---
```

### Step 7 — 完成报告

```
Harness 已进化 ✓

本次更新：
- deploy/SKILL.md: +1 条部署前检查
- project-flow/SKILL.md: +1 条需求澄清问题
- doc/harness-evolution.md: 已记录本次

累计进化次数：N 次（查看 doc/harness-evolution.md 了解历史）
```

## 防退化机制

进化不能让 skill 变得更臃肿、更慢：

1. **写入前先检查**：grep 对应 skill 文件，确认教训不是已有规则的重复
2. **旧规则合并**：如果新教训是对旧规则的强化，**修改旧规则**而不是追加新规则
3. **季度清理**（可选，用户主动触发）：运行 `/evolve --audit`，识别「已知陷阱」里从未被实际踩到的规则，提议删除
4. **每个 skill 的「已知陷阱」不超过 10 条**：超出时合并相似规则或删除最久未触发的

## 长期进化目标追踪

在 `doc/harness-evolution.md` 顶部维护一个滚动统计：

```markdown
# Harness 进化日志

## 总体质量趋势

| 时间段 | 平均重试次数 | 平均用户纠偏 | 首次部署成功率 |
|--------|------------|------------|-------------|
| 2026-04 | 3.2 | 1.8 | 60% |
| 2026-05 | 1.4 | 0.9 | 85% |

**目标**：重试次数 → 0，纠偏次数 → 0，首次部署成功率 → 100%
```

每次 `/evolve` 时更新这个表格（取最近 5 次的平均值）。

## `/evolve --audit` 子命令

用于定期清理过时规则（建议每月一次）：

1. 列出所有 skill 文件的「已知陷阱」条目
2. 对每条，标注：上次被实际触发是什么时候（根据进化日志推断）
3. 超过 90 天未被触发的规则 → 标记为"待删除"，展示给用户确认
4. 用户确认后删除，保持 skill 文件精简

---

## 重要约束

1. **不改逻辑，只加护栏** — `/evolve` 不重构 skill，只加检查/约束/已知陷阱
2. **改之前先读** — 每次修改 skill 文件前必须先 Read 当前内容，避免冲突
3. **用户是最终裁判** — 所有修改必须先展示给用户确认，不静默写入
4. **保持可读性** — 加进去的规则要写给"将来的 AI"看，不是给用户看，所以可以用技术语言，但必须简洁
5. **不覆盖用户手动写的内容** — 如果某 skill 文件有手动注释或特殊说明，只追加，不覆盖
